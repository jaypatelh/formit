Feature Choices

Based off our results from Wizard-of-Oz testing last week, we have decided to proceed with an interface that combines certain aspects of the traditional and gestural UI. The following are a list of features we hope to implement in our final product:

  • Create multiple formations on a canvas view - From our testing last week, we learned that the birds-eye view of a formation was intuitive for users. Since a given piece of choreography can have multiple formations, the ability to create multiple formations is necessary.
  • Save formations for later and notify user appropriately - The formations are created for the purpose of being viewed later. A feature we were missing last week was some type of notification to indicate whether or not a formation was saved, which got extremely confusing for users.
  • When a new formation is created, have the canvas display the previous formation instead of being completely blank. - When users created a new formation, they had to constantly swipe back and forth between formations to remember what dancers were there and where they were. This feature will hopefully make the process run smoother.
  • Affix the list of dancers to the side of the app - The list of dancers is information that is pretty much always relevant in the formation creation process, so there is no need to hide the menu.
  • Create a timeline view - When there are multiple formations and songs going on, it is easy for the user to forget and become confused with where they are chronologically in the set. Having a timeline view allows users to see where they are working, and the view formations within their chronological context.
  • Dot customization with colors and labels - To make visualizing the group easier, the user can label each dot with a dancer name and a unique color representing which choreography “block” that person is in.
  • Swipe gesture to move back and forth between formations - During user testing, this seemed to be an intuitive way to switch between formations because it was a quick gesture and kept the chronology of formations.
  • Adding a “songs” feature to allow users to categorize formations into songs and rearrange the order of songs in a set - This feature will help the user organize the multiple formations involved in a single set. It also makes the process of rearranging the order of songs less complicated, since re-ordering the songs will move the corresponding formations around accordingly. This extra level of organization will also help the user remember what piece of choreography a particular formation is meant for.
  • Creating a grid that dots can lock to - This makes placing and precise alignment of dots easier for the users. That way, they will be less concentrated on the absolute placement and more focused on the relative location.

Implementation Progress

For this first prototype, we wanted to implement the basic features necessary to produce the experience of creating a formation. These are the features available in this version:

  • Set an individual formation (there is no save function yet, so multiple formations can't be created)
  • Create/add dots to the canvas
  • Hold and drag dots to reposition them on the canvas (multi-touch is enabled)
  • Change dot colors to represent separate blocking
  • Create a list of dancers' names
  • Label an individual dot with a dancer's name
  • Select multiple dots at once (coloration to indicate selection)
  • Deleting dots (individual ones or selected groups)

Remaining Implementation

Roughly prioritized
  • Creating/saving multiple formations
  • Adding "songs" feature - organizing formations into songs
  • Timeline view of formations
  • Having previous formation show in new formation
  • Grid for dots to lock onto
  • Swiping between formations

Design Issues

The following is a list of feature that we know we want incorporated into the final product, but we are still ambiguous as to how they should be implemented.

  • How to display the previous formation on a new formation (faded overlay vs. placing dots there)
  • Where timeline should go (navigational breadcrumb that fades in and out vs. affixed to bottom of screen)
  • How to represent the "songs" feature (list view on side vs. coloration vs. table of contents)
